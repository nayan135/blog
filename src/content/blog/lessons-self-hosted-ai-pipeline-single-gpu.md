title: 'What a single GPU actually teaches you about self-hosted AI'

pubDate: '2026-07-03'

Last year I built a pipeline for a client that needed transcription, summarization, and image generation to run without hitting OpenAI's API at all — data residency requirements, the usual story. One box, one RTX 3090 with 24GB of VRAM. That's it. No cluster, no autoscaling, no fallback GPU. Here's what that constraint actually taught me, because it's different from what the tutorials tell you.

The first thing you learn is that VRAM, not compute, is the wall you hit. I loaded Whisper large-v3 for transcription, a 7B Llama model for summarization, and Stable Diffusion for image generation, and figured I'd just run them together since 24GB sounded like a lot. First real request threw `torch.cuda.OutOfMemoryError: CUDA out of memory. Tried to allocate 512.00 MiB`. Whisper alone was fine at ~3GB. The LLM at fp16 wanted 14GB. Stable Diffusion wanted another 4-6GB depending on batch size. Add them up and you're over budget before you've processed a single real user request. On paper the specs looked generous. In practice three models loaded at once left almost nothing for actual inference.

Quantization stopped being an optimization and became the whole strategy. I moved the LLM to GGUF format running through llama.cpp at Q4_K_M, which dropped it from 14GB to under 5GB with a quality hit I genuinely couldn't tell apart in the outputs I was checking. That's the trade nobody tells you about upfront: you don't quantize because you read a benchmark, you quantize because the alternative is your process getting killed by the OOM killer at 2am.

The second lesson was about sequencing, not scaling. With one GPU you can't run things in parallel the way you'd design a system with a team of engineers thinking about throughput. I had to build a queue — I used Redis with a simple worker script — so that transcription, summarization, and image generation ran one at a time, loading and unloading models as needed. That unload/reload cycle is its own tax. Swapping a 5GB quantized model in and out of VRAM took anywhere from 4 to 9 seconds depending on disk I/O, and if you're doing that per request, your latency numbers go from "acceptable" to "why is this so slow" fast. The fix was keeping the most frequently used model resident and only swapping the less common one, which meant redesigning the pipeline order so summarization (the most called step) stayed loaded and image generation, called maybe once per ten requests, took the swap hit.

Third: heat and power are not abstract concerns when you're running this from home or a small office in Kathmandu. NEA load-shedding isn't the constant problem it was a decade ago, but voltage fluctuations still are, and I lost a training run partway through because the wall power dipped and the UPS didn't kick in fast enough. Now everything runs behind a proper online UPS, not the cheap line-interactive kind. The GPU also idles at 45°C and under sustained load in a room without AC it crept past 80°C, which is when I started seeing clock throttling and inference times got noticeably inconsistent — same prompt, same model, but response time varying by almost 2x between requests. A $30 case fan fixed more of my "why is this randomly slow" tickets than any code change did.

The biggest mental shift, though, was accepting that a single-GPU pipeline isn't a smaller version of a cloud pipeline. It's a different system with different failure modes. You stop thinking about horizontal scaling and start thinking about scheduling, memory budgets, and graceful degradation when two requests land at the same second. I ended up writing a simple priority queue where transcription jobs (short, cheap) could jump ahead of image generation jobs (slow, expensive) so the pipeline felt responsive even under load.

None of this is exotic. It's the kind of constraint-driven design you'd have done on a single server in 2012 before cloud elasticity made everyone lazy about resource math. Running AI locally on hardware you can actually see and touch drags you back into that world, and honestly, it made me a better engineer than another few months of calling hosted APIs ever did.
